## TL;DR

Jeśli używasz Meta Conversions API (CAPI) bez parametru `fbc`, tracisz nawet **50% raportowanych konwersji**. Meta nie jest w stanie powiązać zdarzeń z kliknięciami reklam, przez co optymalizacja kampanii działa na niepełnych danych.

W tym artykule pokazuję, jak zdiagnozowałem problem na stronie naszego klienta (Maltaview.pl, apartamenty premium w Poznaniu) i jak rozwiązaliśmy go w 4 krokach: Fluent Forms Pro + Google Tag Manager + n8n.

---

## Czym jest fbc i dlaczego Meta go potrzebuje

`fbc` to **Facebook Click ID**, czyli wartość, którą Meta umieszcza w URL każdej reklamy w momencie kliknięcia. Po kliknięciu reklamy użytkownik trafia na Twoją stronę z parametrem `?fbclid=...` w adresie URL.

Przykład:

```
https://twojastrona.pl/?fbclid=IwAR3xK8YvZ2...
```

Ten identyfikator to **jedyny pewny sygnał**, że dany użytkownik przyszedł z konkretnego kliknięcia reklamy. Gdy wysyłasz zdarzenie (np. `PageView`, `Lead`, `Purchase`) przez Meta CAPI bez `fbc`, Meta próbuje dopasować użytkownika po innych parametrach (IP, User Agent, email, telefon), ale to dopasowanie probabilistyczne, nie deterministyczne.

### Co mówi sama Meta

W panelu Events Manager, w sekcji Jakość zdarzeń, Meta bezpośrednio komunikuje:

> *"Podobni reklamodawcy, którzy przesłali prawidłowy parametr Meta ClickID dla zdarzenia PageView, uzyskali wzrost mediany o co najmniej 100% obecnie raportowanych dodatkowych konwersji."*

Innymi słowy, brak `fbc` może oznaczać, że **widzisz połowę konwersji, które faktycznie się zdarzyły**.

## Event Match Quality (EMQ): jak Meta ocenia jakość danych

Meta ocenia każde zdarzenie w skali 0-10 (EMQ, czyli Event Match Quality). Im wyższa ocena, tym lepiej algorytm optymalizacji rozumie, kto dokonał konwersji.

**Wpływ na EMQ mają m.in.:**

| Parametr | Waga | Uwagi |
|----------|------|-------|
| `fbc` (Facebook Click ID) | Bardzo wysoka | Dopasowanie deterministyczne |
| `fbp` (Facebook Browser ID) | Wysoka | Ciasteczko `_fbp` |
| `em` (email zahashowany) | Wysoka | SHA256 |
| `ph` (telefon zahashowany) | Wysoka | SHA256, format E.164 |
| `external_id` | Średnia | Np. ID klienta w CRM |
| IP + User Agent | Niska | Sygnał pomocniczy |

Na naszej stronie klienta przed poprawką EMQ wynosił **4.4/10**. Meta wyraźnie sugerowała, że główną przyczyną jest brak `fbc`.

## Anatomia problemu: dlaczego to tak łatwo przegapić

### Typowy setup Meta CAPI

Większość wdrożeń Meta CAPI opiera się na schemacie:

1. **Front-end (GTM/Pixel)**, czyli `fbq('track', 'PageView')` po stronie przeglądarki
2. **Back-end (CAPI)**, czyli identyczne zdarzenie wysłane z serwera dla deduplikacji i lepszego dopasowania

Problem pojawia się w punkcie 2. Serwer **nie widzi** URL użytkownika ani jego ciasteczek bezpośrednio. Musi dostać te dane od front-endu. Jeśli front-end ich nie przekazuje, CAPI dostaje zdarzenia **bez `fbc`**.

### Co pokazała diagnostyka

W naszym przypadku (webhook Fluent Forms → n8n → Meta CAPI) struktura payloadu wyglądała tak:

```json
{
  "event_id": "pv_1776690772372_abc123",
  "page_url": "https://maltaview.pl/wybierz-apartament/",
  "fbp": "fb.1.1776158350297.1599172564",
  "fbc": "",
  "user_agent": "Mozilla/5.0..."
}
```

Klucz `fbc` **istniał** w payloadzie, ale zawsze był pustym stringiem. Front-end próbował odczytać ciasteczko `_fbc`, ale **nikt go wcześniej nie zapisywał**.

To jest najczęstszy błąd: developerzy zakładają, że Meta Pixel automatycznie zapisze `_fbc`. To nieprawda. Meta Pixel zapisuje ciasteczko tylko w określonych warunkach (m.in. gdy consent mode pozwala) i tylko w najnowszej wersji Pixela.

## Rozwiązanie krok po kroku

### Krok 1. Przechwyć fbclid z URL i zapisz w ciasteczku `_fbc`

Meta wymaga formatu: `fb.1.{timestamp_ms}.{fbclid}`

```javascript
function captureFbclid() {
  var fbclid = new URLSearchParams(window.location.search).get('fbclid');
  if (fbclid && !getCookie('_fbc')) {
    var fbc = 'fb.1.' + Date.now() + '.' + fbclid;
    setCookie('_fbc', fbc, 90); // waznosc 90 dni, zgodna z Meta attribution window
  }
}
```

Ważne:

- **90 dni**. Dłuższy termin Meta odrzuca jako nieaktualny.
- Zapisuj tylko wtedy, gdy **nie ma** już `_fbc`, żeby nie nadpisać wartości z poprzedniej wizyty z reklamy.

### Krok 2. Dodaj ukryte pole w formularzu

W Fluent Forms Pro (lub dowolnym pluginie formularzy) dodaj **Hidden Field** z nazwą `fbc`. To pole będzie wysłane razem z resztą danych do Twojego backendu.

### Krok 3. Wypełnij pole wartością z ciasteczka

JavaScript na każdej podstronie:

```javascript
function fillFbc() {
  var fbc = getCookie('_fbc');
  if (!fbc) return;
  document.querySelectorAll('input[name="fbc"]').forEach(function(el) {
    el.value = fbc;
  });
}
fillFbc();
document.addEventListener('DOMContentLoaded', fillFbc);
document.addEventListener('focusin', fillFbc);
setInterval(fillFbc, 2000);
```

Cztery sposoby uruchomienia (natychmiast, po DOMContentLoaded, przy focusie, co 2s) to **defensywne programowanie**. Różne pluginy formularzy ładują DOM w różny sposób, a niektóre renderują pola dynamicznie (Elementor, React).

### Krok 4. Obsłuż `fbc` w backendzie CAPI

Na backendzie (u nas: n8n) zbuduj payload do Meta Graph API:

```javascript
const userData = {
  fbp: body.fbp || undefined,
  fbc: body.fbc || undefined,
  client_user_agent: body.user_agent || undefined,
  client_ip_address: clientIp || undefined,
  em: body.email ? sha256(body.email.toLowerCase()) : undefined,
  ph: body.phone ? sha256(normalizePhone(body.phone)) : undefined
};

const capiPayload = {
  data: [{
    event_name: 'Lead',
    event_time: Math.floor(Date.now() / 1000),
    event_id: body.event_id,
    action_source: 'website',
    event_source_url: body.page_url,
    user_data: userData
  }]
};
```

Kluczowa zasada: **każda wartość musi być zahashowana SHA256 PRZED wysłaniem**, czyli email, telefon, external_id. Wyjątkiem są `fbc`, `fbp`, IP i User Agent. Te idą w formie jawnej.

## Jak to zweryfikować

### 1. Weryfikacja front-endu

1. Otwórz stronę w trybie incognito z parametrem: `https://twojastrona.pl/?fbclid=TEST999`
2. F12 → Application → Cookies
3. Powinno pojawić się ciasteczko `_fbc` z wartością `fb.1.{timestamp}.TEST999`

### 2. Weryfikacja backendu

W logach webhooka sprawdź payload przychodzący po wysłaniu formularza. Przed poprawką:

```
fbc: (brak/pusty)
```

Po poprawce:

```
fbc: fb.1.1776690772372.TEST999
```

### 3. Weryfikacja w Meta Events Manager

- **Po 24-48h** `fbc` zacznie pojawiać się w sekcji "Udostępniane parametry".
- **Po ok. 7 dniach** EMQ score powinien wzrosnąć (u nas z 4.4/10 w kierunku 7-8/10).
- Meta pokaże **wzrost raportowanych konwersji**, potencjalnie powyżej 100% wg ich własnych szacunków.

## Consent Mode: nie zapomnij o zgodzie

`_fbc` to **ciasteczko marketingowe** w rozumieniu GDPR. W UE musi być ustawione **po** wyrażeniu zgody marketingowej przez użytkownika.

W GTM oznacza to, że tag zapisujący `_fbc` musi mieć wymaganie zgody `ad_storage`. CookieYes CMP (albo Cookiebot, Complianz) automatycznie zablokuje tag do czasu akceptacji cookies marketingowych.

**Konsekwencje:**

- Użytkownicy, którzy **akceptują** cookies, dostają pełny `fbc`, co daje wysoką jakość atrybucji.
- Użytkownicy, którzy **odrzucają** cookies, nie mają `fbc`, CAPI dalej działa, ale z niższym EMQ.

To jest trade-off, który trzeba zaakceptować. Obchodzenie consent mode to naruszenie GDPR i może skutkować karami.

## Najczęstsze pułapki

### 1. Meta Pixel nie zapisuje `_fbc` automatycznie we wszystkich wypadkach

Wbrew częstemu założeniu, `_fbc` jest zapisywane przez Meta Pixel tylko w określonych warunkach. Najlepszą praktyką jest **zawsze** zaimplementować własny zapis z URL.

### 2. Nieprawidłowy format `fbc`

Meta wymaga formatu: `fb.1.{timestamp_ms}.{fbclid}`

Częste błędy:

- Użycie `Date.now() / 1000` (sekundy zamiast milisekund). CAPI odrzuci.
- Brak prefiksu `fb.1.`. CAPI odrzuci.
- Użycie samego `fbclid` zamiast pełnego formatu `fbc`.

### 3. Wygasłe `fbc`

Meta attribution window to **7 dni dla kliknięć**, ale samo ciasteczko `_fbc` ma 90 dni ważności. W n8n warto dodać walidację: jeśli timestamp z `fbc` jest starszy niż 90 dni, pomiń parametr.

```javascript
function isValidFbc(fbc) {
  var parts = fbc.split('.');
  if (parts.length < 4) return false;
  var ts = parseInt(parts[2], 10);
  var ninetyDaysMs = 90 * 24 * 60 * 60 * 1000;
  return (Date.now() - ts) < ninetyDaysMs;
}
```

### 4. Form plugin nadpisuje Hidden Field

Niektóre pluginy (zwłaszcza w trybie AJAX) mogą resetować wartości pól przed wysłaniem. Stąd `setInterval(fillFbc, 2000)` w naszym kodzie, czyli brute-force ponowne wypełnianie co 2 sekundy.

## Podsumowanie

Brak `fbc` w Meta CAPI to cichy zabójca wyników kampanii reklamowych. Nie powoduje błędów, nie pojawia się w alertach, zdarzenia docierają do Mety pozornie prawidłowo, ale algorytm optymalizacji pracuje na połowie danych.

Wdrożenie jest technicznie proste (4 kroki, ok. 30 minut pracy), ale wymaga zrozumienia całego łańcucha: front-end → cookies → formularz → backend → Meta Graph API. Każde ogniwo musi być poprawnie skonfigurowane.

Jeśli prowadzisz kampanie reklamowe na Meta i używasz CAPI, **sprawdź swój EMQ w Events Manager jeszcze dziś**. Jeśli jest poniżej 7, prawdopodobnie tracisz pieniądze na nieoptymalnie zoptymalizowanych kampaniach.
