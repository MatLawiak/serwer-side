## Problem: piksel Meta nie wystarcza

Jeśli prowadzisz kampanie Meta Ads i polegasz wyłącznie na pikselu po stronie przeglądarki, tracisz dane. To nie teoria — to fakt potwierdzony przez samą Meta w panelu Events Manager, gdzie możesz zobaczyć procent zdarzeń „nie pokrytych przez API konwersji".

Dla większości firm liczba ta wynosi 30–60%. Co to oznacza w praktyce? Algorytm Meta optymalizuje kampanie na podstawie niepełnego obrazu. Płacisz za wyniki, których część jest niewidoczna.

## Klient: maltaview.pl — deweloper nieruchomości

Maltaview to projekt deweloperski premium. Model biznesowy oparty na leadach — potencjalny nabywca wypełnia formularz kontaktowy, handlowiec dzwoni i umawia prezentację apartamentu. Lejek jest długi, a wartość każdego leadu wysoka.

**Sytuacja przed wdrożeniem:**
- Kampanie Meta Ads optymalizowane pod zdarzenie Lead
- Piksel Meta działał poprawnie po stronie przeglądarki
- Brak Conversions API — wszystkie dane z przeglądarki
- EMQ (Event Match Quality) nieznana, brak serwera

**Data wdrożenia:** 26 marca 2026

## Architektura którą wybraliśmy

Zamiast standardowego sGTM z tagiem Meta CAPI, zdecydowaliśmy się na **bezpośrednią integrację przez n8n** — narzędzie do automatyzacji uruchomione na własnym serwerze.

Dlaczego? Bo daje nam:
- Pełną kontrolę nad danymi i logiką przetwarzania
- Wbudowany error handling z powiadomieniami email
- Możliwość integracji z CRM, Notion, Slackiem
- Łatwy debugging przez panel wykonań
- Zero zależności od zewnętrznych SaaS

### Schemat działania

```
Przeglądarka użytkownika
    │
    ├── Meta Pixel (browser) ──────────────────► Meta
    │     [event_id dla deduplikacji]
    │
    └── fetch() z GTM ──► n8n webhook ──────────► Meta CAPI
          [PageView + cookies]     [enriched data]
```

**Formularz kontaktowy:**
```
Fluent Forms
    └──► n8n webhook ──► Meta CAPI Lead
                    └──► GA4 Measurement Protocol
```

### Co wysyłamy do Meta CAPI

Każde zdarzenie PageView zawiera:

| Parametr | Źródło | Pokrycie |
|---|---|---|
| Adres IP | nagłówek x-real-ip | 92% |
| User Agent | przeglądarka | 100% |
| fbp (ID przeglądarki) | cookie _fbp | 93% |
| fbc (ID kliknięcia) | cookie _fbc | 16% * |
| external_id | SHA-256(IP+UA) | 100% |

*fbc jest niskie bo to normalny odsetek ruchu z reklam Meta — cookie istnieje tylko u użytkowników którzy kliknęli reklamę.

Każdy Lead dodatkowo zawiera:
- Email (SHA-256)
- Telefon (SHA-256, znormalizowany do formatu +48)
- Imię i nazwisko (SHA-256)
- external_id (SHA-256 z emaila — najstabilniejszy identyfikator)

### Deduplikacja

Kluczowy element poprawnej implementacji dual-tracking (piksel + CAPI). Każde zdarzenie generuje unikalny `event_id`:

```javascript
// PageView
const eventId = 'pv_' + Date.now() + '_' + Math.random().toString(36).slice(2,8);
fbq('track', 'PageView', {}, {eventID: eventId});
// Ten sam eventId leci do n8n → CAPI
```

Meta automatycznie deduplikuje zdarzenia z tym samym `event_id` — nie ma podwójnego liczenia.

## Wyniki po 4 dniach

**EMQ (Event Match Quality): 5.4/10**

To wynik po zaledwie 4 dniach od wdrożenia, gdy Meta jeszcze przetwarza dane. Dla porównania — sam piksel bez CAPI osiąga typowo 3–5/10.

**Co Meta zgłasza jako dostępne ulepszenia:**
- Dodanie `external_id` → +16.15% raportowanych konwersji
- Dodanie emaila → +2.69%
- Zmiana IPv4 na IPv6 → dodatkowe dopasowania

W dniu pisania tego artykułu wdrożyliśmy external_id w obu workflowach (PageView i Lead). EMQ powinno wzrosnąć do 7+ w ciągu 7 dni.

**Integracja:** `Nie wykryto problemów. W czasie rzeczywistym` ✓

## Zewnętrzny_id — dlaczego jest kluczowy

external_id to parametr który pozwala Meta łączyć zdarzenia od tej samej osoby nawet gdy cookies są wyczyszczone lub użytkownik zmienia urządzenie.

**Dla PageView** generujemy go jako hash z IP + User Agent — deterministyczny identyfikator urządzenia:
```javascript
const externalId = sha256((clientIp + ':' + userAgent).toLowerCase());
```

**Dla Lead** używamy hasha emaila (najbardziej stabilny identyfikator):
```javascript
const externalId = sha256(email.toLowerCase().trim());
```

Dzięki temu gdy ta sama osoba wejdzie na stronę (PageView z external_id opartym o IP/UA) a potem wypełni formularz (Lead z external_id opartym o email) — Meta może próbować połączyć te zdarzenia w jedną ścieżkę konwersji.

## Co ważne: to nie jest substytut sGTM dla Google

Podejście n8n działa doskonale dla Meta CAPI, ale **nie zastępuje sGTM dla Google Ads**. Google Ads wymaga danych płynących przez własną infrastrukturę (GA4 → sGTM → Enhanced Conversions). N8n obsługuje Meta, sGTM obsługuje Google — razem tworzą kompletny setup.

## Wnioski dla lead generation w nieruchomościach

Branża nieruchomości ma specyficzne cechy które sprawiają, że server-side tracking jest tu szczególnie wartościowy:

1. **Długi lejek zakupowy** — użytkownik może kliknąć reklamę i wypełnić formularz po tygodniach. Bez serwera i 1st-party cookies atrybucja przepada.

2. **Wysoka wartość leadu** — jeden lead to potencjalnie kilka milionów złotych transakcji. Każdy "utracony" lead to realna strata w optymalizacji algorytmu.

3. **Formularz jako jedyna konwersja** — w e-commerce możesz śledzić wiele mikro-konwersji. Tu masz jeden kluczowy event: wypełnienie formularza. Musi być śledzony bezbłędnie.

4. **Offline conversion potential** — w przyszłości można śledzić przez GA4 Measurement Protocol dalsze etapy lejka: kwalifikacja leadu, prezentacja, rezerwacja, akt notarialny.
